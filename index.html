VM Environment Setup
on macOS with VMware

Before writing a single command, you need a safe, isolated Linux environment. We'll build one from scratch on your MacBook using VMware Fusion — the industry standard for macOS virtualization.

⏱ ~45 min🧩 Hands-on setup📋 Pre-requisite for all modules

What is Virtualization?

A Virtual Machine (VM) is a software-based computer that runs inside your real computer. The software that creates and manages VMs is called a hypervisor. Your Mac acts as the host; the Linux VM is the guest. The hypervisor intercepts hardware calls from the guest and translates them safely, with near-native performance.

Type 1 — Bare Metal Hypervisor

Runs directly on hardware with no host OS layer. Used in data centers for maximum efficiency. Examples: VMware ESXi, Microsoft Hyper-V, Xen, KVM.

Type 2 — Hosted Hypervisor

Runs as an application on top of an existing OS (your macOS). Easier to use. Examples: VMware Fusion, VirtualBox, Parallels Desktop.

💡
Why VM over dual-boot? A crashed Linux install cannot harm your Mac. You can snapshot, clone, and reset in seconds. Network is isolated. It's the standard approach for learning, development, and testing.

The Linux Architecture Stack

Linux is organized in concentric rings. Each layer communicates only with adjacent layers through well-defined interfaces:

┌──────────────────────────────────────────────────┐ │ USER SPACE │ │ ┌────────────────────────────────────────────┐ │ │ │ Applications (Firefox, vim, git, bash) │ │ │ ├────────────────────────────────────────────┤ │ │ │ Shell (bash, zsh) — command interpreter │ │ │ ├────────────────────────────────────────────┤ │ │ │ C Standard Library (glibc) — libc wrappers│ │ │ └──────────────────┬─────────────────────────┘ │ │ │ System Calls (syscalls) │ ├─────────────────────▼────────────────────────────┤ │ KERNEL SPACE │ │ ┌───────────┐ ┌───────────┐ ┌────────────────┐ │ │ │ Process │ │ Memory │ │ Virtual File │ │ │ │ Scheduler │ │ Manager │ │ System (VFS) │ │ │ ├───────────┤ ├───────────┤ ├────────────────┤ │ │ │ Network │ │ Device │ │ Security (LSM) │ │ │ │ Stack │ │ Drivers │ │ SELinux/AppArmor│ │ │ └───────────┘ └───────────┘ └────────────────┘ │ ├──────────────────────────────────────────────────┤ │ HARDWARE CPU · RAM · Disks · Network · GPU │ └──────────────────────────────────────────────────┘

Kernel Space vs User Space

Kernel space has unrestricted access to all hardware. A bug here can freeze or corrupt the entire system. User space processes run in protected memory — a crash only kills that one process. System calls are the controlled bridge: when your program reads a file, it calls read() in glibc, which triggers a syscall instruction, switches CPU privilege level, and the kernel handles the actual disk I/O before returning.

SubsystemResponsibility
Process Scheduler (CFS)Decides which process runs on which CPU core, for how long. Uses Completely Fair Scheduler algorithm.
Virtual Memory ManagerGives each process the illusion of exclusive 64-bit address space. Manages physical RAM, page tables, swap.
VFS (Virtual File System)Abstraction making ext4, xfs, nfs, procfs, and tmpfs all appear as a single unified tree.
Network StackFull TCP/IP implementation from Ethernet (L2) to sockets (L4). Netfilter hooks for iptables/nftables.
Device DriversKernel modules that translate hardware-specific operations into standardized kernel APIs.

The Boot Process

From power button to login prompt, each stage hands control to the next:

1. BIOS/UEFI Power-on self-test, finds bootable disk, loads MBR/EFI partition ↓ 2. GRUB2 Bootloader: presents OS selection menu, loads kernel + initramfs into RAM ↓ 3. Kernel init Decompresses itself, detects hardware, initializes memory management ↓ 4. initramfs Temporary root filesystem with drivers needed to mount the real root partition ↓ 5. systemd (PID 1) Real init: starts all services in parallel based on dependency graph ↓ 6. Login Display manager (GDM) or virtual TTY ready for user login

Kernel Modules

Rather than compiling every driver into the kernel, Linux loads modules dynamically — like plugins. This keeps the kernel lean and allows drivers to be loaded/unloaded at runtime.

Kernel module management

I/O Redirection

Every process inherits three standard file descriptors: stdin (fd 0), stdout (fd 1), stderr (fd 2). You can redirect any of them:

Redirection operators
$ ls -la > list.txt # stdout → file (OVERWRITES) $ echo "line" >> log.txt # stdout → file (APPENDS) $ wc -l < list.txt # file → stdin $ ls /nope 2> errors.txt # stderr → file $ ls /nope 2>> errors.txt # stderr → file (append) $ make 2>&1 | tee build.log # stderr → stdout, pipe both to tee $ cmd > /dev/null 2>&1 # discard ALL output (silence)   # Here-document: multi-line stdin $ cat << EOF > config.txt host=localhost port=5432 EOF

Pipes and logical operators

Filesystem Hierarchy
Standard (FHS)

The Linux filesystem follows a well-defined standard. Knowing where things live is essential for configuration, security hardening, and troubleshooting.

⏱ ~50 min

The FHS Directory Tree

/ ← root of everything ├── bin Essential user commands (ls, cp, mv) — symlink → /usr/bin ├── boot Kernel images, initramfs, GRUB config ├── dev Device files: /dev/sda (disk), /dev/null, /dev/random ├── etc System-wide config files (plain text, human-readable) ├── home User home directories: /home/alice, /home/bob ├── lib Shared libraries for /bin and /sbin ├── media Mount points for removable media (USB drives, CD-ROMs) ├── mnt Temporary mount point for manual disk mounts ├── opt Optional third-party software (not managed by apt) ├── proc Virtual FS: live kernel & process info (NOT real disk files) ├── root Home directory for the root superuser ├── run Runtime data: PIDs, sockets — tmpfs, wiped on reboot ├── sbin System binaries for root (fdisk, iptables) — symlink → /usr/sbin ├── sys Virtual FS: hardware & driver info exposed by kernel ├── tmp Temporary files — world-writable, cleared on reboot ├── usr Most user programs and libraries │ ├── bin Primary user commands (python3, git, vim, curl) │ ├── lib Shared libraries for /usr/bin programs │ ├── local Locally compiled programs (/usr/local/bin, /usr/local/lib) │ └── share Architecture-independent data (docs, icons, man pages) └── var Variable data that grows over time ├── log System and application logs ← you'll spend time here ├── www Web server document roots ├── lib Persistent app state (databases, package lists) └── spool Print queues, cron job spools

Virtual Filesystems: /proc and /sys

These directories contain no real files on disk. They are interfaces the kernel exposes so userspace programs can read (and sometimes write) kernel data structures:

Exploring /proc and /sys
# System-wide kernel info $ cat /proc/version # full kernel version string $ cat /proc/uptime # seconds since boot $ cat /proc/loadavg # 1/5/15 min load averages + process counts $ cat /proc/net/dev # per-interface byte/packet counters   # Per-process directory (replace $$ with any PID) $ ls /proc/$$ # current shell's /proc directory $ cat /proc/$$/status # name, state, UIDs, memory $ cat /proc/$$/cmdline | tr '\0' ' ' # command that started this process   # /sys — modify kernel parameters (be careful) $ cat /sys/class/net/ens33/address # MAC address of interface $ cat /sys/block/sda/size # disk size in 512-byte sectors

Hard Links vs Symbolic Links

Hard Link

A second directory entry pointing to the same inode on disk. Same data, two names. Deleting one doesn't remove data until all hard links are gone. Cannot cross filesystem boundaries.

ln file.txt hardlink.txt

Symbolic Link (Symlink)

A file that contains a path to another file — like a Windows shortcut. Can cross filesystems. If the target is deleted, the symlink becomes a dangling broken link.

ln -s /path/to/target link.txt
Inodes and links
$ ln file.txt hard.txt # create hard link $ ln -s /etc/hosts hosts-link # create symlink $ ls -li # -i shows inode number 12345 -rw-r--r-- 2 alice alice 100 Apr 2 file.txt ← same inode! 12345 -rw-r--r-- 2 alice alice 100 Apr 2 hard.txt ← same inode! 12346 lrwxrwxrwx 1 alice alice 9 Apr 2 hosts-link -> /etc/hosts   $ stat file.txt # detailed inode info: size, perms, times $ find / -inum 12345 2>/dev/null # find all hard links to inode 12345

Permissions, ACLs
& Special Bits

Linux file permissions are a three-tier access control system. Special bits add setuid/setgid/sticky behavior. ACLs extend permissions to individual users and groups.

⏱ ~70 min🔒 Security critical

Reading Permission Strings

Output of: ls -la
- rwx │ r-x │ r-- 1 alice devteam 4096 Apr 2 script.sh
File type: -=regular file  |  d=directory  |  l=symlink  |  c=char device  |  b=block device
Owner (alice): rwx = read(4) + write(2) + execute(1) = 7
Group (devteam): r-x = read(4) + execute(1) = 5
Others: r-- = read(4) only = 4 → permission string: chmod 754

chmod — Numeric and Symbolic

chmod reference
# Numeric (octal) — r=4, w=2, x=1, add them up per group chmod 755 script.sh # rwxr-xr-x standard executable chmod 644 data.txt # rw-r--r-- standard data file chmod 600 id_rsa # rw------- SSH private key (SSH REQUIRES this!) chmod 700 ~/.ssh/ # rwx------ SSH dir (SSH REQUIRES this!) chmod 640 /etc/secret # rw-r----- root:root, readable by group   # Symbolic — more readable for targeted changes chmod u+x script.sh # add execute for user/owner chmod g-w file.txt # remove write from group chmod o= file.txt # remove all permissions from others chmod a+r file.txt # add read for all (a = u+g+o) chmod ug=rw,o= file # set owner+group to rw, others to nothing   # Recursive chmod -R 755 /var/www/html/ # set 755 on all files AND dirs # BETTER: different perms for files vs dirs: $ find /var/www -type f -exec chmod 644 {} \; $ find /var/www -type d -exec chmod 755 {} \;

Special Permission Bits

BitOctalEffect on FileEffect on Directory
SUID (SetUID)4000Runs as file owner regardless of who executes. e.g. /usr/bin/passwd runs as root so it can write to /etc/shadow.Ignored
SGID (SetGID)2000Runs as file's groupNew files created inside inherit the directory's group. Essential for shared project folders.
Sticky Bit1000No modern effectOnly the file owner (and root) can delete files, even if others have write access to the directory. Used on /tmp.
Special bits in practice
# SUID — lowercase 's' in owner execute position $ ls -l /usr/bin/passwd -rwsr-xr-x 1 root root 59976 Mar 15 /usr/bin/passwd   # Sticky bit on /tmp — 't' in others execute position $ ls -ld /tmp drwxrwxrwt 14 root root 4096 Apr 2 /tmp   # Set special bits (numeric: add 4/2/1 as leading digit) $ chmod 4755 /usr/local/bin/myapp # SUID + rwxr-xr-x $ chmod g+s /shared/project/ # SGID on directory $ chmod +t /shared/uploads/ # sticky bit   # Find all SUID files (security audit) $ find / -perm /4000 -type f 2>/dev/null

umask — Default Permission Mask

umask defines which permission bits are removed from newly created files and directories. It's a subtract-from mask:

umask
$ umask # show current umask (usually 0022) 0022 # Files: 666 - 022 = 644 (rw-r--r--) # Dirs: 777 - 022 = 755 (rwxr-xr-x)   $ umask 027 # stricter: others get no permissions # Files: 666 - 027 = 640 (rw-r-----) # Set permanently in ~/.bashrc or /etc/profile

Access Control Lists (ACLs)

Standard permissions only handle one owner, one group, and others. ACLs let you set permissions for specific users or groups on a per-file basis:

getfacl / setfacl
$ sudo apt install acl   # View ACLs on a file $ getfacl report.txt user::rw- # owner group::r-- # group other::--- # others   # Grant specific user access $ setfacl -m u:bob:rw report.txt $ setfacl -m g:devteam:rx /shared/ # group access   # Default ACL: applies to new files created inside directory $ setfacl -d -m g:devteam:rw /project/   # Remove ACL entry $ setfacl -x u:bob report.txt $ setfacl -b report.txt # remove ALL ACL entries   # Files with ACLs show + in ls output $ ls -la report.txt -rw-rw-r--+ 1 alice alice 1234 Apr 2 report.txt

How SSH Cryptography Works

SSH uses asymmetric (public-key) cryptography. You generate a key pair: the private key stays on your machine forever and is never transmitted. The public key is placed on servers. Authentication works via a challenge-response: the server sends a challenge encrypted with your public key; only your private key can decrypt it.

Basic SSH usage
$ ssh alice@192.168.1.100 $ ssh -p 2222 alice@server.example.com # non-default port $ ssh -v alice@server # verbose — debugging auth issues   # Run a single command remotely (no interactive shell) $ ssh alice@server "df -h && uptime" $ ssh alice@server "sudo systemctl restart nginx"   # Copy files — scp (simple) $ scp file.txt alice@server:/tmp/ # upload $ scp alice@server:/var/log/app.log . # download $ scp -r folder/ alice@server:/opt/ # recursive   # rsync — better than scp for large transfers (only sends changes) $ rsync -avz ./local/ alice@server:/remote/ $ rsync -avz --delete ./local/ alice@server:/remote/ # mirror (deletes extras) $ rsync -avz --dry-run ./local/ alice@server:/remote/ # preview only

Port forwarding

Real Script: System Health Check

health_check.sh
#!/usr/bin/env bash set -euo pipefail   WARN_DISK=80 WARN_LOAD=4 SERVICES=(ssh nginx cron)   check_disk() { local usage usage=$(df / | awk 'NR==2 {gsub(/%/,"",$5); print $5}') if (( usage >= WARN_DISK )); then echo "⚠ DISK: ${usage}% used on /" else echo "✓ Disk: ${usage}% used" fi }   check_services() { for svc in "${SERVICES[@]}"; do if systemctl is-active --quiet "$svc"; then echo "✓ $svc: running" else echo "✗ $svc: NOT running" fi done }   check_load() { local load; load=$(cut -d' ' -f1 /proc/loadavg) if (( $(echo "$load >= $WARN_LOAD" | bc -l) )); then echo "⚠ Load: $load (high)" else echo "✓ Load: $load" fi }   echo "=== Health Check $(hostname) @ $(date) ===" check_disk check_load check_services

Cron environment is minimal. Cron doesn't load your ~/.bashrc or ~/.bash_profile. Always use full paths in cron jobs (/usr/bin/python3 not just python3). Add PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin at the top of your crontab.

systemd Timers — The Modern Way

systemd timers are more powerful than cron: they log to journald, can have dependencies, and run missed jobs on next boot with Persistent=true:

Creating a systemd timer
# /etc/systemd/system/backup.service [Unit] Description=Daily Backup Job   [Service] Type=oneshot # runs once and exits User=alice ExecStart=/home/alice/scripts/backup.sh   # /etc/systemd/system/backup.timer [Unit] Description=Run backup.service daily at 3 AM   [Timer] OnCalendar=*-*-* 03:00:00 # daily at 3 AM Persistent=true # run if missed (machine was off) RandomizedDelaySec=300 # jitter ±5min to avoid thundering herd   [Install] WantedBy=timers.target   $ sudo systemctl daemon-reload $ sudo systemctl enable --now backup.timer $ systemctl list-timers # see all timers and next run time $ journalctl -u backup.service # view job output/logs
